# Test 4: Query Rewriting (IN vs EXISTS vs JOIN)

### Test Objective

The objective of this experiment is to compare three different SQL approaches that answer a similar business question:

"Find customers who have placed at least one order."

The goal is to evaluate:

- How PostgreSQL executes each query.
- Whether query formulation affects performance.
- The execution plans generated by the optimizer.
- Which approach is most efficient for existence checks.

---

### Query 1: EXISTS

```sql
EXPLAIN ANALYZE
SELECT *
FROM customers c
WHERE EXISTS (
    SELECT 1
    FROM orders o
    WHERE o.customer_id = c.customer_id
);
```

### Analysis

Execution flow:

1. Scan all orders.
2. Aggregate unique customer IDs.
3. Build a hash table.
4. Scan customers.
5. Match customer IDs using the hash table.

---

### Query 2: IN

```sql
EXPLAIN ANALYZE
SELECT *
FROM customers
WHERE customer_id IN (
    SELECT customer_id
    FROM orders
);
```

### Result

<img src="../docs/screenshots/test4/IN.png" width="800">

### Metrics

|Metric	|Value|
|---|---|
|Execution Plan	|Hash Join
|Planning Time	|0.241 ms
|Execution Time	|163.001 ms
|Rows Returned	|166,717
|Hash Memory Usage	|909 KB|

---

### Analysis

Execution flow:

1. Filter customers from Bangalore.
2. Build hash table.
3. Scan orders.
4. Match rows through hash lookups.

This plan required less intermediate processing compared to the other approaches.

---

### Query 3: JOIN + DISTINCT

```sql
EXPLAIN ANALYZE
SELECT DISTINCT c.*
FROM customers c
JOIN orders o
ON c.customer_id = o.customer_id;
```

### Result

<img src="../docs/screenshots/test4/JOIN.png" width="800">

### Metrics

|Metric	|Value|
|---|---|
|Execution Plan	|Parallel Hash Join + Sort + Unique|
|Planning Time	|0.528 ms|
|Execution Time	|751.671 ms|
|Rows Returned	|99,998|
|Sort Method	|External Merge|
|Temporary Disk Usage	|~20–30 MB per worker|

---

### Analysis

Execution flow:

1. Join all matching customer-order pairs.
2. Produce approximately one million intermediate rows.
3. Sort all rows.
4. Remove duplicates using Unique.

The sorting phase became the dominant cost.

### Performance Improvements

#### IN vs EXISTS

$$
\frac{286.724}{163.001} \approx 1.76
$$

Result: **IN is ~1.76x faster than EXISTS.**

#### IN vs JOIN

$$
\frac{751.671}{163.001} \approx 4.61
$$

Result: **IN is ~4.61x faster than JOIN.**

#### EXISTS vs JOIN

$$
\frac{751.671}{286.724} \approx 2.62
$$

Result: **EXISTS is ~2.62x faster than JOIN.**

---

### Key Findings

1. **Query formulation significantly impacts performance.**
    - Different SQL statements can generate very different execution plans.
2. **The IN query produced the fastest execution time in this experiment.**
    - PostgreSQL generated an efficient Hash Join plan.
3. **JOIN + DISTINCT was the slowest approach.**
    - Large intermediate result sets required expensive sorting and deduplication.
4. **Disk-based sorting dramatically increased execution time.**
    - External merge sorting introduced additional I/O overhead.
5. **PostgreSQL's optimizer rewrites logically equivalent queries differently.**
    - Understanding execution plans is often more important than the SQL syntax itself.
