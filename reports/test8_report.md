# Test 8: Full Index vs Partial Index

**Note**:<br>
In previous experiments, indexes were created on all rows of a table.

However, many real-world applications frequently query only a specific subset of data. Creating indexes for every row may introduce unnecessary maintenance overhead when only certain records are regularly accessed.

PostgreSQL provides Partial Indexes, which index only rows matching a specified condition.

### Objectives

- Understand how Partial Indexes work.
- Compare Full Index and Partial Index performance.
- Observe PostgreSQL's index selection behavior.
- Analyze execution plans and buffer usage.
- Evaluate query performance for filtered workloads.

---

### No Index

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT *
FROM orders
WHERE customer_id = 5000
AND status = 'DELIVERED';
```

### Result

<img src="../docs/screenshots/test8/no_index.png" width="800">

### Metrics

|Metric|Value|
|---|---|
|Scan Type|Bitmap Heap Scan|
|Execution Time|0.700 ms|
|Planning Time|0.489 ms|
|Shared Hits|13|
|Shared Reads|1|
|Rows Returned|0|

### Analysis:

- The entire table is scanned.
- A large number of rows are filtered.
- Query performance depends heavily on table size.
- Resource consumption is high.

---

### Full Index

Create Index:
```sql
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);
```

Update statistics:

```sql
ANALYZE orders;
```

Re-run:
```sql
EXPLAIN (ANALYZE, BUFFERS) 
SELECT * 
FROM orders 
WHERE customer_id = 5000 
AND status = 'DELIVERED';
```

### Result

<img src="../docs/screenshots/test8/full_index.png" width="800">

### Metrics

|Metric|Value|
|---|---|
|Scan Type|Bitmap Heap Scan|
|Execution Time|0.067 ms|
|Planning Time|0.146 ms|
|Shared Hits|3|
|Shared Reads|0|
|Rows Returned|0|

### Analysis

- PostgreSQL uses the composite index.
```sql
customer_id = 5000 AND status = 'DELIVERED'
```
- Buffer usage decreases significantly.
- Query execution becomes much faster.

---

### Partial Index

Remove Full Index:
```sql
DROP INDEX idx_orders_customer_status;
```

Create Partial Index:
```sql
CREATE INDEX idx_delivered_orders 
ON orders(customer_id) 
WHERE status = 'DELIVERED';
```

Update statistics:
```sql
ANALYZE orders;
```

Re-run:
```sql
EXPLAIN (ANALYZE, BUFFERS) 
SELECT * 
FROM orders 
WHERE customer_id = 5000 
AND status = 'DELIVERED';
```

### Result

<img src="../docs/screenshots/test8/partial_index.png" width="800">

### Metrics

|Metric	|Value|
|---|---|
| Scan Type	| Bitmap Heap Scan|
| Execution Time	| 0.059 ms|
| Planning Time	| 0.220 ms|
| Shared Hits	| 3|
| Shared Reads	| 0|
| Rows Returned	| 0|

### Analysis

- PostgreSQL uses the Partial Index.
```sql
status = 'DELIVERED'
```
- Only rows satisfying the index condition are considered.
- The execution plan remains highly efficient.
- Buffer activity remains low.

---

### Benchmark

| Metric | No Index | Full Index | Partial Index |
|---|---|---|---|
| Scan Type | Bitmap Heap Scan | Bitmap Heap Scan | Bitmap Heap Scan |
| Execution Time | 0.700 ms | 0.067 ms | 0.059 ms |

| Shared Reads | 1 | 0 | 0 |
| Rows Returned | 0 | 0 | 0 |

**Note**: `Rows returned = 0` for all cases, since the `customer_id = 5000` has **no order** with `status = 'DELIVERED'` in the orders table.

---

### Performance Comparision

Metric Used: `Execution Time`

**Full Index vs No Index**

$$
\frac{0.700}{0.067} \approx 10.45
$$

Result: Full Index is **~10.45x** faster than No Index

**Partial Index vs No Index**

$$
\frac{0.700}{0.059} \approx 11.86
$$

Result: Partial Index is **~11.86x** faster than No Index

**Partial Index vs Full Index**

$$
\frac{0.067}{0.059} \approx 1.13
$$

Result: Partial Index is **~1.13x** faster than Full Index

---

### Key Findings

- PostgreSQL use Partial Indexes when the query condition matches the index predicate. 
- Partial Indexes significantly improve query performance compared to scanning the entire table.
- Execution plans generated by Partial Indexes are often similar to those produced by Full Indexes.
- Buffer activity decreases substantially when PostgreSQL can use an appropriate index.
- Partial Indexes are particularly useful when queries frequently target a specific subset of data.
