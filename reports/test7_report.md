# Test 7: Machine-Readable Execution Plans with EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)

**Note**: <br>
Unlike Tests 1–6, this report does not benchmark an optimization technique. The value of Test 7 lies in understanding how PostgreSQL exposes execution plans as structured data for automation and analysis.<br>
In previous experiments, execution plans were analyzed using PostgreSQL's default text-based format.

While textual execution plans are useful for manual inspection, they are difficult to process programmatically. PostgreSQL provides an alternative JSON format that represents execution plans as structured data.

### Objective

- Generate execution plans in JSON format.
- Understand the structure of PostgreSQL plan trees.
- Identify important plan attributes such as node type, row counts, execution time, and buffer statistics.
- Compare textual and JSON execution plans.

---

### Query

**Textual Execution Plan**
```sql
EXPLAIN ( 
    ANALYZE, 
    BUFFERS, 
) 
SELECT * 
FROM orders 
WHERE customer_id = 300;
```

---

**JSON Execution Plan**
```sql
EXPLAIN ( 
    ANALYZE, 
    BUFFERS, 
    FORMAT JSON 
) 
SELECT * 
FROM orders 
WHERE customer_id = 300;
```

### JSON Result

```json
{
    "Planning Time": 0.117,
    "Execution Time": 0.087,
    "Node Type": "Bitmap Index Scan",
    "Relation Name": "orders",
    "Actual Rows": 15.00,
    "Shared Hit Blocks": 3,
    "Shared Read Blocks": 0
}
```

The above JSON execution plan displays only required fields(e.g, execution plan, node type etc.)

For complete result: View **[Complete Plan](./plans/complete_json_plan.json)**

### Observations

- PostgreSQL returns the execution plan as a structured JSON document.
- Plan nodes are represented as JSON objects.
- Metrics are stored as key-value pairs.
- The same information available in the text plan is preserved.

---

### Important JSON Fields

**Node Type**

Indicates the operation performed by PostgreSQL.

Examples:
```
- Seq Scan
- Index Scan
- Bitmap Heap Scan
- Bitmap Index Scan
- Hash Join
```

---

**Relation Name**

Identifies the table being accessed.

Example:
```
orders
```

---

**Actual Rows**

Represents the number of rows returned by a plan node.

Example:
```
15
```

---

**Planning Time**

Time spent creating the execution plan.

Example:
```
0.117 ms
```

---

**Execution Time**

Time spent executing the query.

Example:
```
0.087 ms
```

---

**Buffer Statistics**

```
Shared Hit Blocks
Shared Read Blocks
```

These values describe the amount of memory and disk activity generated by the query.

---

### Why JSON Plans Matter

Many PostgreSQL performance tools internally use JSON execution plans instead of text plans.

Examples include:

- Query analyzers
- Performance monitoring platforms
- SQL optimization tools
- Database observability systems

JSON plans make it possible to:

- Parse execution plans automatically.
- Detect inefficient scan types.
- Track performance trends.
- Generate optimization recommendations.

---

### Key Findings

- PostgreSQL can generate execution plans in structured JSON format.
- JSON plans contain the same information available in text execution plans.
- Plan nodes, row counts, execution times, and buffer statistics can be accessed programmatically.
- JSON execution plans provide the foundation for automation and performance tooling.
- Machine-readable execution plans are essential for large-scale query analysis and monitoring systems.
